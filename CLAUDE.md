# Flume — プロジェクトメモリ

macOS ネイティブのストリーミング・ワークフローエンジン(個人探究プロジェクト)。
設計の全体像・ドメイン語彙・設計判断(ADR)・フェーズ計画は **DESIGN.md** が正。

## セッション開始時に必ずやること

1. `.claude/memory/MEMORY.md` を読む(セッション横断の記憶の索引)
2. `journal/` の最新エントリを読む(前回どこまで進み、次の一手が何か)

## 体制(重要)

- **コードはユーザー(yosuke)が書く。** このプロジェクトはリハビリと探究を兼ねている
- Claude の役割: 設計相談、コードレビュー、詰まったときのヒント、n8n との答え合わせの伴走
- **頼まれない限り実装コードを書かない。** 書くのはドキュメント・ジャーナル・レビューコメントまで
- ヒントを求められたら、答えを直接書くのではなく方向を示す(シグネチャ、参照すべき API、考え方)

## 開発の流儀(DESIGN.md より)

- 設計判断は「予想 → 実装 → n8n で答え合わせ」のサイクル。答え合わせ結果は DESIGN.md の ADR に記録
- 仕様の本体はゴールデンテスト(prose より test)。`swift test` が常に回る状態を維持
- n8n のソースは**自分の設計を決めた後**にしか読まない

## ジャーナル運用

- セッション終了時(またはユーザーが「記録して」と言ったとき)に `journal/YYYY-MM-DD.md` を追記・作成
- テンプレート: **やったこと / 決めたこと / 学び・発見 / 次回の一手**
- 「次回の一手」は具体的なアクションで書く(次セッションはこれを読んで即着手できること)

## メモリ運用

- セッション横断で残すべき事実(設計の暗黙前提、ユーザーの好み、外部リンク)は `.claude/memory/` に 1 ファイル 1 事実で保存し、`MEMORY.md` に索引を追記
- 進捗そのものはジャーナルへ、設計判断は DESIGN.md の ADR へ(メモリには書かない)

## ビルド・検証

```bash
swift build          # ビルド
swift test           # テスト(仕様の本体)
```
